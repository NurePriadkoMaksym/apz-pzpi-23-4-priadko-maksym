3 ОПИС ВИКОНАНОЇ РОБОТИ

3.1 Загальні відомості про систему
Назва системи: Netflix.
Розробник: Netflix Inc. (США).
Призначення системи: глобальна потокова платформа для перегляду відеоконтенту (фільми, серіали, шоу) через Інтернет. Система забезпечує доставку мультимедійного контенту мільйонам користувачів у реальному часі з адаптацією якості відео під мережеві умови.
Користувачі: глядачі, які використовують веббраузери, мобільні додатки, Smart TV та інші пристрої.
Основні функції:
• перегляд відео в режимі streaming; 
• персоналізовані рекомендації контенту; 
• пошук і фільтрація; 
• управління профілями; 
• обробка платежів; 
• адаптивна доставка відео (adaptive streaming).
3.2 Архітектурний стиль системи
Система Netflix побудована на основі мікросервісної архітектури, що поєднує клієнт-серверну модель взаємодії з розподіленою хмарною інфраструктурою та подієво-орієнтованими механізмами обробки даних. У межах цієї архітектури функціональність системи декомпозована на велику кількість незалежних сервісів, кожен з яких відповідає за окрему бізнес-функцію та може розгортатися і масштабуватися автономно. Такий підхід дозволяє забезпечити гнучкість системи та її здатність ефективно обробляти значні обсяги запитів від користувачів у глобальному масштабі.
Ключовим викликом для систем такого класу є забезпечення стабільної доставки відеоконтенту з мінімальними затримками та високою якістю обслуговування при значних коливаннях навантаження. Для вирішення цього завдання розробники використали хмарну платформу та розподілену інфраструктуру доставки контенту (CDN), що дозволяє наближати дані до кінцевого користувача. Додатково застосування подієво-орієнтованого підходу забезпечує асинхронну обробку подій, таких як дії користувача або оновлення рекомендацій, що зменшує навантаження на критичні компоненти системи. Водночас розділення функціональності на окремі мікросервіси дозволяє ізолювати ресурсоємні процеси, наприклад обробку відео або генерацію рекомендацій, що забезпечує ефективне масштабування системи та підвищує її відмовостійкість у разі пікових навантажень.
3.3 Основні компоненти системи
Система складається з таких підсистем:
1. Інтерфейс користувача (Frontend)
* Клієнтські застосунки:
* Web (React); 
* мобільні додатки (iOS, Android); 
* Smart TV. 
* Функції:
* відображення контенту; 
* взаємодія з користувачем; 
* відправка запитів до серверів. 
2. Серверна частина (Backend)
Складається з великої кількості мікросервісів, написаних переважно на Java. Основні сервіси:
* User Service; 
* Recommendation Service; 
* Playback Service; 
* Billing Service. 
3. Рівень бізнес-логіки
Реалізується всередині мікросервісів. 
Включає:
* алгоритми рекомендацій; 
* обробку запитів користувача; 
* логіку доступу до контенту; 
* персоналізацію. 
4. Рівень зберігання даних
Використовуються різні типи баз даних:
* NoSQL (Cassandra) — для великих обсягів даних; 
* MySQL — для транзакцій; 
* кеш (Redis, EVCache). 
5. Допоміжні сервіси
* CDN (Content Delivery Network) - Open Connect; 
* система моніторингу; 
* система логування; 
* балансувальники навантаження. 
3.4 Взаємодія компонентів
Взаємодія в системі Netflix має розподілений характер та поєднує синхронні й асинхронні механізми обміну даними. Стандартні, некритичні до часу операції, такі як авторизація, отримання списків контенту або управління профілем, виконуються через HTTP/REST-запити. Клієнт звертається до API Gateway, який маршрутизує запити до відповідних мікросервісів, що обробляють їх і повертають результат.
Натомість процес відтворення відео реалізується через мережу доставки контенту (CDN), яка забезпечує ефективну передачу потокових даних безпосередньо користувачу. Передача відео здійснюється за принципом адаптивного стримінгу, що дозволяє динамічно змінювати якість залежно від умов мережі.
Синхронізація між компонентами забезпечується подієво-орієнтованим підходом: дії користувача генерують події, які передаються між сервісами через брокери повідомлень і використовуються для аналітики та персоналізації. Такий підхід дозволяє уникнути жорсткої зв’язаності компонентів і підвищує масштабованість системи.
На рисунку 3.1 відображено контейнерну архітектуру системи Netflix. Показано клієнтську частину, центральні мікросервіси (User Service, Playback Service, Media Service, Authentication Service), а також допоміжні сервіси аналітики та обробки даних. Окремо представлено сховища даних (MySQL, Cassandra) та CDN для доставки відеоконтенту.

3.5 Масштабованість та надійність
Завдяки мікросервісній архітектурі система Netflix підтримує ефективне горизонтальне масштабування. Балансувальники навантаження, інтегровані у хмарну інфраструктуру, динамічно розподіляють трафік між численними інстансами сервісів, що дозволяє обробляти значні обсяги запитів користувачів та автоматично масштабувати систему при пікових навантаженнях.
На рівні клієнтських застосунків використовуються механізми кешування та адаптивного стримінгу, що дозволяють зменшити затримки та забезпечити безперервне відтворення відео навіть при нестабільному мережевому з’єднанні. Основні бекенд-сервіси побудовані за принципом stateless, що дає можливість легко перенаправляти запити між різними інстансами без втрати стану користувача.
Для забезпечення відмовостійкості застосовуються механізми ізоляції сервісів та шаблон Circuit Breaker, який запобігає каскадним відмовам у разі недоступності окремих компонентів. Додатково використовується підхід Chaos Engineering, що дозволяє тестувати систему в умовах штучно створених збоїв і підвищувати її надійність.
Кешування реалізовано за допомогою розподілених систем, що зменшують навантаження на основні бази даних і прискорюють обробку запитів. Для збереження даних застосовується реплікація та географічне розподілення, що забезпечує резервування інформації та доступність сервісу навіть у разі відмови окремих дата-центрів.
Платформа повністю інтегрована з хмарною інфраструктурою, зокрема сервісами AWS, що забезпечує автоматичне масштабування, гнучкість розгортання та високу доступність системи у глобальному масштабі.

3.6 Приклади програмної реалізації
В основі архітектури Netflix лежить мікросервісна та подієво-орієнтована парадигма програмування. Наведені нижче словесні представлення з посиланням на фрагменти коду у додатках (псевдокод на основі реальної архітектури, мови Java та JavaScript) демонструють ключові принципи побудови системи: обробку REST-запитів, подієву взаємодію між сервісами та роботу клієнтського інтерфейсу.
3.6.1 Обробка REST API запитів (див. дод. В.1)
Архітектурний принцип: Service-Oriented Architecture & Stateless Services. Використання REST API дозволяє мікросервісам взаємодіяти між собою та з клієнтом через стандартизовані HTTP-запити, при цьому кожен запит обробляється незалежно без збереження стану на сервері.
3.6.2 Подієва взаємодія між сервісами (див. дод. В.2)
Архітектурний принцип: Event-Driven Architecture. Використання брокерів повідомлень дозволяє сервісам асинхронно обмінюватися подіями, що забезпечує слабку зв’язаність компонентів та високу масштабованість системи.
3.6.3 Клієнтський запит та відображення даних (див. дод. В.3)
Архітектурний принцип: Client-Server Interaction & Declarative UI. Клієнтський застосунок отримує дані через API та відображає їх у вигляді інтерфейсу, динамічно оновлюючи контент без перезавантаження сторінки.
