2 ЗАВДАННЯ
На цій лабораторній роботі потрібно розробити мобільний програмний застосунок для будь-якої поширеної на сьогодні мобільної платформи. Мобільний застосунок повинен бути частиною системи і повинен взаємодіяти з нею. Для реалізації програмного застосунку можна використовувати технології для реалізації програмного забезпечення для мобільних платформ, вказані лектором, а саме Kotlin для Android та Swift для iOS. Звіт повинен містити опис прийнятих інженерних рішень. У ньому повинна бути показана будова мобільного застосунку та його взаємодія з іншими частинами системи.


3 ОПИС ВИКОНАНОЇ РОБОТИ

3.1 Загальні відомості про застосунок
Мобільний застосунок PythonWiki призначений для навчання користувачів основам Python через систему статей, курсів, персонального прогресу та графа знань. Застосунок реалізовано для Android з використанням мови Kotlin та декларативного UI-фреймворку Jetpack Compose.
Основна ідея системи полягає в тому, що користувач авторизується у застосунку, отримує доступ до навчальних матеріалів, читає статті, накопичує XP, переходить між ролями та може переглядати зв’язки між темами через інтерактивну карту знань.
Застосунок є клієнтською частиною більшої системи та взаємодіє з backend-сервісом через HTTP API. Backend відповідає за авторизацію, збереження користувачів, статей, курсів, прогресу та структури графа знань.
3.2 Основні функції застосунку
Метою розробки мобільного застосунку є створення зручного інтерфейсу для доступу до навчальної платформи PythonWiki з мобільного пристрою.
Основні функції застосунку:
* реєстрація нового користувача;
* вхід у систему;
* перегляд головного екрана з рекомендованими матеріалами;
* перегляд бібліотеки статей;
* пошук та фільтрація статей;
* перегляд курсів;
* відкриття та читання статей;
* оновлення прогресу користувача після читання;
* перегляд профілю користувача;
* перегляд графа знань;
3.3 Загальна архітектура застосунку
Система складається з таких підсистем:
1. UI-шар
UI реалізовано за допомогою Jetpack Compose. Екрани застосунку розміщені у пакеті ui/screens, спільні компоненти — у ui/components, а інтерактивна карта знань — у ui/graph.
2. Шар стану застосунку
Файл AppState.kt містить опис основного стану UI: поточна сесія, завантажений контент, режим авторизації, повідомлення про помилки, активний процес відкриття статті та інформація про зміну прогресу.
3. Repository-шар
Файл PythonWikiRepository.kt інкапсулює логіку роботи з backend API, локальним збереженням сесії, авторизацією, отриманням профілю, статей, курсів і графа знань.
4. Модель даних
Файл Models.kt містить data-класи, які описують основні сутності мобільного застосунку: користувача, статтю, курс, граф, сесію, dashboard та прогрес.
3.4 Екрани застосунку
Екрани реалізовані у файлах AuthScreens.kt та ContentScreens.kt.
Екран авторизації
LoginScreen підтримує два режими:
• вхід у систему;
• реєстрація нового користувача.
Користувач вводить email, пароль, а у режимі реєстрації також username. Після натискання кнопки відповідний callback передає дані до PythonWikiApp, який викликає метод login() або register() у repository.
Головний екран
HomeScreen відображає:
• інформацію з dashboard;
• рекомендовані статті;
• spotlight-статтю;
• останні курси;
• кнопку оновлення даних.
Бібліотека
LibraryScreen реалізує логіку пошуку та фільтрації статей. Користувач може шукати статті за назвою або категорією, а також фільтрувати їх за доступністю:
• усі статті;
• доступні;
• заблоковані.
Курси
CoursesScreen відображає список курсів та пов’язані зі статтями матеріали. Кожен курс містить список articleIds, за якими UI знаходить відповідні статті у загальному списку.
Граф знань
GraphScreen відображає статистику графа, центральні теми та інтерактивну карту знань. Для візуалізації використовується компонент KnowledgeMap.
Профіль
ProfileScreen показує:
• username;
• email;
• роль користувача;
• кількість XP;
• XP до наступної ролі;
• кількість завершених статей;
• 
• кнопки оновлення та виходу із системи.
Екран статті
ArticleDetailScreen відповідає за перегляд повного тексту статті. Після відкриття статті застосунок отримує повний контент з backend і показує повідомлення про зміну прогресу, наприклад отримані XP або перехід на нову роль.
3.5 Взаємодія мобільного застосунку з іншими частинами системи
Мобільний застосунок взаємодіє з backend-сервером через HTTP-запити. Уся мережева логіка зосереджена в класі PythonWikiRepository.
Основні API-запити:

Для захищених запитів використовується JWT-токен, який передається у HTTP-заголовку:
Authorization: Bearer <token>
Після успішної авторизації токен, refresh token, user id, username та роль користувача зберігаються локально у SharedPreferences. Завдяки цьому застосунок може автоматично відновити сесію після повторного запуску.
3.4 Опис прийнятих інженерних рішень
3.4.1 Централізований стан застосунку
Стан застосунку винесено у AppUiState. Це дозволяє зберігати в одному місці:
* активну сесію;
* контент;
* помилки;
* режим авторизації;
* стан завантаження;
* інформацію про оновлення прогресу.
Централізований state підвищує передбачуваність роботи UI та зменшує дублювання логіки між екранами.
3.4.2 Локальне збереження сесії
Для збереження токена та даних останнього входу використано SharedPreferences. Це дозволяє:
* не змушувати користувача входити повторно після кожного запуску;
* автоматично відновлювати попередню сесію;
* підставляти останні email/password у форму входу.
Разом з тим, збереження пароля у SharedPreferences є компромісним рішенням. Для production-версії доцільніше використовувати більш захищений механізм, наприклад EncryptedSharedPreferences.
3.4.3 Асинхронна робота через coroutines
Усі операції, пов’язані з мережею та локальним сховищем, виконуються асинхронно. У repository використовується Dispatchers.IO, а у UI - rememberCoroutineScope. Це дозволяє не блокувати головний потік Android-застосунку та забезпечує плавну роботу інтерфейсу.
3.4.4 Repository pattern
Для роботи з API створено клас PythonWikiRepository. Він приховує від UI деталі:
* HTTP-запитів;
* JSON-парсингу;
* збереження сесії;
* обробки помилок;
* побудови моделей даних;
* обробки GraphML.
Завдяки цьому UI не залежить напряму від реалізації мережевого шару.

В.2 Генерація фрагмента коду для обробки GraphML
Назва інструменту та версія: ChatGPT.
Мета використання: отримання прикладу Kotlin-коду для парсингу GraphML-даних, які використовуються для побудови графа знань у мобільному застосунку.
Внесок автора: код було адаптовано під моделі проєкту PythonWiki, додано використання GraphConnection, обробку вузлів node і зв’язків edge, а також фільтрацію дублікатів вузлів через distinct().
Промпт:
«Напиши приклад Kotlin-функції для Android, яка приймає GraphML у вигляді рядка, парсить XML, знаходить усі node та edge елементи, і повертає список id вузлів та список зв’язків між ними.
Результат:
1   private fun parseGraphMl(graphMl: String): ParsedGraph {
2       val parser = XmlPullParserFactory.newInstance().newPullParser().apply {
3           setFeature(XmlPullParser.FEATURE_PROCESS_NAMESPACES, true)
4           setInput(StringReader(graphMl))
5       }
6       val nodeIds = mutableListOf()
7       val connections = mutableListOf()
8
9       while (parser.eventType != XmlPullParser.END_DOCUMENT) {
10          if (parser.eventType == XmlPullParser.START_TAG) {
11              when (parser.name) {
12                  "node" -> parser.getAttributeValue(null, "id")
13                      ?.toIntOrNull()
14                      ?.let(nodeIds::add)
15                  "edge" -> {
16                      val fromId = parser.getAttributeValue(null, "source")?.toIntOrNull()
17                      val toId = parser.getAttributeValue(null, "target")?.toIntOrNull()
18                      if (fromId != null && toId != null) {
19                          connections += GraphConnection(fromId = fromId, toId = toId)
20                      }
21                  }
22              }
23          }
24          parser.next()
25      }
26
27      return ParsedGraph(
28          nodeIds = nodeIds.distinct(),
29          connections = connections
30      )
31  }
